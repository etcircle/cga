# CGA v1.2 P1b — CALLS-flush lookup indexes, proving run

- **Date:** 2026-05-21
- **Task:** v1.2 P1b — cut the `flush_inscope` sub-phase, which the v1.2 P0.5
  profile pinned at 81.6 % of `function_calls` on di-copilot and ~89 % on
  cgc-fork.
- **Lever shipped:** c.iii (indexes). Lever c.ii (fallback-bucket split) was
  implemented, falsified by proving, and discarded — see below.

## The investigation

The eng review picked **c.ii** — split the polymorphic `_INSCOPE_BATCH_CYPHER`
into an exact lane and a name-only fallback lane so the common case stops
paying for the `OPTIONAL MATCH (fbF:Function {name: ...}) WHERE exact IS NULL`
clause. c.ii was implemented in full (four buffers, two extra templates,
Python pre-classification, 32 passing tests, byte-parity preserved).

**c.ii delivered zero speedup.** cgc-fork per-lane timing:

| lane | per-spec | template |
|---|---:|---|
| in-scope exact (fbF-free) | 1.071 ms/spec | new |
| in-scope fallback (polymorphic) | 1.511 ms/spec | v1.1 |

The exact template *was* faster per spec — but the aggregate `flush_inscope`
did not move (9.32 s → 9.31 s), because P1a's blended 1.278 ms/spec was
already the weighted average of exact-hit specs (~1.07 ms — FalkorDB's
planner skips the `fbF` `OPTIONAL MATCH` when `WHERE exact IS NULL` fails)
and exact-miss specs (~1.51 ms — `fbF` genuinely fires). **FalkorDB already
short-circuits the `fbF` clause.** The eng review's premise — "the planner is
paying the cost of optionality" — was wrong. c.ii was discarded rather than
shipping ~150 lines of correct-but-useless complexity.

Re-profiling `flush_inscope` then found the actual cost. A prefix of the
in-scope query — caller resolution plus `RETURN count(caller)` — *timed out*
against the post-index cgc-fork graph. A query doing strictly less than the
full template should not be slower. `EXPLAIN` showed why: every
`OPTIONAL MATCH (n:Function {name, path})` resolved with a
**`Node By Label Scan`** — a full scan of every `Function` node, O(graph
size) per call site. That is the 15× per-file blowup from cgc-fork to
di-copilot the P0.5 profile recorded, and it is why c.ii could not help: the
`fbF` clause was never the cost; the unindexed scans the *other four*
`OPTIONAL MATCH`es do were.

Root cause is in `create_schema()`. It wrapped every statement in a single
`try` block. The first statement is a `CREATE CONSTRAINT` that — once CGA's
FalkorDB session wrapper rewrites it — FalkorDB rejects. It raised, the lone
`except` swallowed it, and **every subsequent statement was skipped.** CGA
had been cold-indexing with no indexes at all.

A direct before/after probe on one captured 2000-spec in-scope chunk
isolated the lever — explicit `CREATE INDEX` statements on the lookup keys,
run against the populated graph:

```
BEFORE indexes:  2701 ms  (1350.55 µs/spec)   EXPLAIN: Node By Label Scan ×5
AFTER  indexes:   129 ms  (  64.56 µs/spec)   EXPLAIN: Node By Index Scan ×5
                  ─────────────────────────
                  20.9× faster, lookups only
```

## What changed

`create_schema()` is restructured so each statement runs in its own `try` —
a rejected statement can no longer abort the rest. That single change
restores every index the method was meant to create.

How the CALLS-flush lookup keys end up indexed (see the comments in
`create_schema` and `database.py::_translate_schema_query`):

- **Function / Class / Interface / Variable / Parameter** — their
  `CREATE CONSTRAINT ... REQUIRE (name, path, ...) IS UNIQUE` statements are
  composite-key. CGA's session wrapper downgrades a composite-key constraint
  to a composite `CREATE INDEX` on `(name, path, ...)`. FalkorDB resolves the
  `{name, path}` and name-only `OPTIONAL MATCH`es in the flush templates from
  a leading-column prefix of that composite index. These constraints had
  *never run* before — the single-`try` block aborted on the first
  (single-key) constraint, well before reaching them.
- **File** — its constraint is single-key (`f.path IS UNIQUE`), which the
  wrapper rewrites to a form FalkorDB still rejects. File is the one
  CALLS-flush label (`_FILESCOPE_BATCH_CYPHER` resolves callers by
  `File.path`) that gets no index from the constraint path, so P1b adds an
  explicit `CREATE INDEX file_path`.

The pre-v1.2 `function_lang` / `class_lang` range indexes — also skipped by
the old abort — now get created too. The `code_search_index` full-text index
does **not**: the session wrapper rewrites `CREATE FULLTEXT INDEX` to a
`RETURN 1` no-op (a pre-existing limitation; out of P1b scope).

Uniqueness is still not enforced — the composite constraints became plain
indexes, the single-key ones create nothing. Engine writes MERGE on the full
key so duplicates do not arise; porting the constraints to FalkorDB's
`GRAPH.CONSTRAINT` form is a follow-up.

The graph schema (node and edge shape, `docs/v1.0-graph-schema.md`) is
unchanged: an index is a storage structure, not schema. Query templates, the
CALLS buffers, `add_file_to_graph`, and the MCP surface are untouched.

## Proving run — `cgc-fork`

Same setup as v1.0–v1.2 P1a: FalkorDB at `127.0.0.1:6379`, fresh graph,
`CGA_PROFILE_PHASES=true`.

| metric            | v1.0   | v1.1   | P0+P0.5 | P1a     | **P1b**     |
|-------------------|-------:|-------:|--------:|--------:|------------:|
| wall time         | 65.1 s | 41.5 s | 32.54 s | 23.25 s | **11.60 s** |
| `function_calls`  | —      | —      | 10.33 s | 10.41 s | **0.75 s**  |
| `flush_inscope`   | —      | —      | 9.55 s  | 9.32 s  | **0.41 s**  |
| `flush_filescope` | —      | —      | 0.79 s  | 0.84 s  | **0.04 s**  |
| `add_file_to_graph` | —    | —      | 7.94 s  | 2.46 s  | **0.66 s**  |

```
=== cold-index phase profile (cgc-fork, v1.2 P1b) ===
  files indexed: 166
  wall time:     11.60 s
  parse_file                 6.58 s   56.7%   39.64 ms/file
  add_file_to_graph          0.66 s    5.7%    3.96 ms/file
  minimal_file_node          1.88 s   16.2%
  function_calls             0.75 s    6.5%    4.52 ms/file
    └─ resolve                0.30 s   40.4% of function_calls
    └─ flush_inscope          0.41 s   54.2% of function_calls
    └─ flush_filescope        0.04 s    5.0% of function_calls
```

**Correctness — pass.** Every edge count derived from parseable source is
byte-identical to v1.0 / v1.1: `CALLS=4795`, `INHERITS=21`, `IMPORTS=691`,
`Function=1494`, `Class=182`, `CONTAINS=8042`, `HAS_PARAMETER=1831`. An index
is a read-path structure — it cannot change which nodes or edges a MERGE
produces. PASS gate #3 met.

**Speed.** `flush_inscope` 9.32 s → 0.41 s (−96 %, ~23× — matching the
isolated 20.9× probe). `function_calls` −93 %. `add_file_to_graph` −73 % on
top of P1a, because its per-file MERGEs hit the same indexes. Total
cold-index wall is −50 % vs P1a and **−82 % vs the v1.0 baseline**.
`flush_inscope` is now 54.2 % of `function_calls` — **PASS gate #7 (target
< 60 % on cgc-fork) met**.

## Proving run — `di-copilot`

Fresh cold index of di-copilot (4 356 files), `CGA_PROFILE_PHASES=true`.

| metric            | v1.1 baseline | v1.2 P0.5 | v1.2 P1a | **v1.2 P1b** |
|-------------------|--------------:|----------:|---------:|-------------:|
| wall time         | 4 687.6 s     | 4 134.0 s | 3 249 s  | **331.8 s**  |
| `function_calls`  | —             | 2 803 s   | 2 169 s  | **16.2 s**   |
| `flush_inscope`   | —             | 2 288 s   | 1 782 s  | **7.1 s**    |
| `add_file_to_graph` | —           | 987 s     | 755 s    | **18.1 s**   |

**di-copilot cold index: 69 min → 5.5 min (−92 %).**

    └─ flush_filescope        1.52 s    9.4% of function_calls

**Correctness — pass.** All 11 node/edge counts byte-identical to the
v1.0 / v1.1 / P0.5 baseline: `File=4 356`, `Function=19 085`, `Class=3 134`,
`Module=2 870`, `Repository=1`, `Directory=606`, `CONTAINS=103 599`,
`CALLS=91 459`, `INHERITS=79`, `IMPORTS=17 215`, `HAS_PARAMETER=20 661`.
**PASS gate #4 met.**

**Speed — this is the real prize.** `flush_inscope` 2 288 s → 7.1 s
(−99.7 %, ~320×). The unindexed label scan is O(graph size); di-copilot has
12.8× more `Function` nodes than cgc-fork, so the BEFORE per-spec cost was
~21× cgc-fork's, while the index lookup is ~flat — exactly why the di-copilot
win dwarfs cgc-fork's. The result is *not* run-to-run variance: `parse_file`
(CPU-bound, untouched) held at 68.68 ms/file vs P0.5's 69.37, the stable
anchor that proves the DB-phase collapse is the index change, not a faster
machine. `add_file_to_graph` 987 s → 18 s for the same reason — its MERGEs
key on the same indexes. Cold index drops from 69 min to 5.5 min; the
dominant phase is now `parse_file` (CPU-bound tree-sitter, 61.5 %).

## Test coverage

31 tests pass (30 from v1.1 + P1a, plus one new). `ruff` clean.

- `test_v1_2_p1b_schema_creates_callsflush_lookup_indexes` — asserts
  `db.indexes()` leaves `Function`/`Class` `(name, path)` and `File(path)`
  range-indexed after `create_schema()`. It checks the *outcome*, not the
  mechanism, so it catches a regression whether the cause is a dropped
  index, a return of the single-`try` abort, or a translation-layer change.

The existing 30 tests are the byte-parity guard: every CALLS / CONTAINS /
IMPORTS / HAS_PARAMETER assertion must still pass with indexes present.

## What's unchanged

- Graph schema (`docs/v1.0-graph-schema.md`) — node/edge shape frozen.
- Query core, the CALLS flush templates, `add_file_to_graph`, the four
  v1.0 MCP tools, ADR-0002.
- `_INSCOPE_BATCH_CYPHER` / `_FILESCOPE_BATCH_CYPHER` — byte-for-byte the
  v1.1 templates. P1b changed only what indexes exist underneath them.

## Follow-ups

- Port the `CREATE CONSTRAINT` statements to FalkorDB's `GRAPH.CONSTRAINT`
  syntax so uniqueness is actually enforced (currently relies on MERGE keys).
- `parse_file` is now the largest cold-index phase (56.7 % cgc-fork, 61.5 %
  di-copilot). Tree-sitter parsing is CPU-bound; parallelism is the obvious
  lever for a future version.

## Verdict

**Ship.** v1.2 P1b lands the real `flush_inscope` lever — the lookup indexes
`create_schema()` had silently been failing to create since v1.0:

- cgc-fork: `flush_inscope` −96 %, `function_calls` −93 %, total wall −50 %
  vs P1a and −82 % vs v1.0.
- di-copilot: `flush_inscope` −99.7 %, cold index 69 min → 5.5 min (−92 %).
  Every node and edge count byte-identical to v1.0 / v1.1.
- 31 tests pass; `ruff` clean; schema, query core, MCP surface untouched.

c.ii (the eng review's pick) was implemented, measured, falsified, and
discarded — recorded here so it is not re-attempted. The proving gate did
its job twice: it caught a dead lever before it shipped, and it confirmed
the real one against both proving repos before this PR.
