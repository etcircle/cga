# CGA v1.2 P1b — CALLS-flush lookup indexes, proving run

- **Date:** 2026-05-21
- **Task:** v1.2 P1b — cut the `flush_inscope` sub-phase, which the v1.2 P0.5
  profile pinned at 81.6 % of `function_calls` on di-copilot and ~89 % on
  cgc-fork.
- **Lever shipped:** c.iii (indexes). Lever c.ii (fallback-bucket split) was
  implemented, falsified by proving, and discarded — see below.

## The investigation

The eng review picked **c.ii** — split the polymorphic `_INSCOPE_BATCH_CYPHER`
into an exact lane and a name-only fallback lane so the common case stops
paying for the `OPTIONAL MATCH (fbF:Function {name: ...}) WHERE exact IS NULL`
clause. c.ii was implemented in full (four buffers, two extra templates,
Python pre-classification, 32 passing tests, byte-parity preserved).

**c.ii delivered zero speedup.** cgc-fork per-lane timing:

| lane | per-spec | template |
|---|---:|---|
| in-scope exact (fbF-free) | 1.071 ms/spec | new |
| in-scope fallback (polymorphic) | 1.511 ms/spec | v1.1 |

The exact template *was* faster per spec — but the aggregate `flush_inscope`
did not move (9.32 s → 9.31 s), because P1a's blended 1.278 ms/spec was
already the weighted average of exact-hit specs (~1.07 ms — FalkorDB's
planner skips the `fbF` `OPTIONAL MATCH` when `WHERE exact IS NULL` fails)
and exact-miss specs (~1.51 ms — `fbF` genuinely fires). **FalkorDB already
short-circuits the `fbF` clause.** The eng review's premise — "the planner is
paying the cost of optionality" — was wrong. c.ii was discarded rather than
shipping ~150 lines of correct-but-useless complexity.

Re-profiling `flush_inscope` then found the actual cost. `GraphBuilder`'s
`create_schema()` wrapped every statement in a single `try` block; the first
statement is a Neo4j-syntax `CREATE CONSTRAINT` that FalkorDB rejects
("Invalid constraint command, use the GRAPH.CONSTRAINT command instead"). It
raised, the lone `except` swallowed it, and **every subsequent statement —
including all the `CREATE INDEX` statements — was skipped.** CGA had been
cold-indexing with no indexes at all.

So every `OPTIONAL MATCH (n:Function {name, path})` in the flush templates
resolved with a full `Node By Label Scan` — O(graph size) per call site.
That is the 15× per-file blowup from cgc-fork to di-copilot the P0.5 profile
recorded, and it is why c.ii could not help: the `fbF` clause was never the
cost; the unindexed scans the *other four* `OPTIONAL MATCH`es do were.

A direct before/after probe on one captured 2000-spec in-scope chunk:

```
BEFORE indexes:  2701 ms  (1350.55 µs/spec)   EXPLAIN: Node By Label Scan ×5
AFTER  indexes:   129 ms  (  64.56 µs/spec)   EXPLAIN: Node By Index Scan ×5
                  ─────────────────────────
                  20.9× faster, lookups only
```

## What changed

`create_schema()` is restructured so each statement runs in its own `try` —
a rejected constraint can no longer abort index creation. Five range indexes
are added, backing every `(name, path)` lookup the CALLS flush and the
add-file MERGEs key on:

- `Function(name)`, `Function(path)`
- `Class(name)`, `Class(path)`
- `File(path)`

The pre-v1.2 `function_lang` / `class_lang` indexes and the `code_search_index`
full-text index — which the single-`try` block had also been silently
skipping since v1.0 — now get created too.

The graph schema (node and edge shape, `docs/v1.0-graph-schema.md`) is
unchanged: an index is a storage structure, not schema. Query templates,
the CALLS buffers, `add_file_to_graph`, and the MCP surface are untouched.

The Neo4j-syntax `CREATE CONSTRAINT` statements are kept as a record of the
intended uniqueness keys. They still fail (one compact warning instead of an
abort); porting them to FalkorDB's `GRAPH.CONSTRAINT` form is a follow-up.
Uniqueness is not enforced today and was not before — engine writes MERGE on
the full key, so duplicates do not arise in practice.

## Proving run — `cgc-fork`

Same setup as v1.0–v1.2 P1a: FalkorDB at `127.0.0.1:6379`, fresh graph,
`CGA_PROFILE_PHASES=true`.

| metric            | v1.0   | v1.1   | P0+P0.5 | P1a     | **P1b**     |
|-------------------|-------:|-------:|--------:|--------:|------------:|
| wall time         | 65.1 s | 41.5 s | 32.54 s | 23.25 s | **11.60 s** |
| `function_calls`  | —      | —      | 10.33 s | 10.41 s | **0.75 s**  |
| `flush_inscope`   | —      | —      | 9.55 s  | 9.32 s  | **0.41 s**  |
| `flush_filescope` | —      | —      | 0.79 s  | 0.84 s  | **0.04 s**  |
| `add_file_to_graph` | —    | —      | 7.94 s  | 2.46 s  | **0.66 s**  |

`function_calls` phase profile (P1b):

```
=== cold-index phase profile (cgc-fork, v1.2 P1b) ===
  files indexed: 166
  wall time:     11.60 s
  repository_node            0.00 s    0.0%
  discovery                  0.34 s    2.9%
  pre_scan_imports           1.27 s   10.9%
  parse_file                 6.58 s   56.7%   39.64 ms/file
  add_file_to_graph          0.66 s    5.7%    3.96 ms/file
  minimal_file_node          1.88 s   16.2%
  inheritance_links          0.05 s    0.4%
  function_calls             0.75 s    6.5%    4.52 ms/file
    └─ resolve                0.30 s   40.4% of function_calls
    └─ flush_inscope          0.41 s   54.2% of function_calls
    └─ flush_filescope        0.04 s    5.0% of function_calls
  unaccounted                0.07 s    0.6%
```

**Correctness verdict — pass.** Every edge count derived from parseable
source is byte-identical to v1.0 / v1.1: `CALLS=4795`, `INHERITS=21`,
`IMPORTS=691`, `Function=1494`, `Class=182`, `CONTAINS=8042`,
`HAS_PARAMETER=1831`. An index is a read-path structure — it cannot change
which nodes or edges a MERGE produces. PASS gate #3 met.

**Speed verdict.** `flush_inscope` dropped 9.32 s → 0.41 s (−96 %, ~23× —
matching the isolated 20.9× probe). `function_calls` dropped 93 %.
`add_file_to_graph` dropped a further 73 % on top of P1a, because its
per-file `MATCH (f:File {path})` / `MERGE (n:Function {name, path,
line_number})` MERGEs hit the same new indexes. Total cold-index wall is
down 50 % vs P1a and **82 % vs the v1.0 baseline**. `flush_inscope` is now
54.2 % of `function_calls` — **PASS gate #7 (target < 60 % on cgc-fork) met**.

The dominant phase is now `parse_file` (tree-sitter, 56.7 %, CPU-bound,
unchanged at ~40 ms/file) — a different problem for a future version.

## Proving run — `di-copilot`

_Pending. The di-copilot proving (`CGA_PROFILE_PHASES=true`) is in flight as
this doc lands. The unindexed label scan is O(graph size); di-copilot has
12.8× more `Function` nodes than cgc-fork, so the BEFORE per-spec cost was
~21× cgc-fork's while the index lookup is ~flat — the scale win should be
larger than cgc-fork's. Byte-parity assertion on `CALLS=91 459` /
`IMPORTS=17 215` / `Function=19 085` / `Class=3 134` / `CONTAINS=103 599` /
`HAS_PARAMETER=20 661` etc. (PASS gate #4). Numbers folded in on completion._

## Test coverage

31 tests pass (30 from v1.1 + P1a, plus one new). `ruff` clean.

- `test_v1_2_p1b_schema_creates_callsflush_lookup_indexes` — asserts
  `db.indexes()` registers all five `(label, property)` pairs after
  `create_schema()`. A future change that drops an index, or a FalkorDB
  version that stops accepting the `CREATE INDEX` form, fails this test
  instead of hiding as a slow proving run.

The existing 30 tests are the byte-parity guard: every CALLS / CONTAINS /
IMPORTS / HAS_PARAMETER assertion must still pass with indexes present, and
does.

## What's unchanged

- Graph schema (`docs/v1.0-graph-schema.md`) — node/edge shape frozen.
- Query core, the CALLS flush templates, `add_file_to_graph`, the four
  v1.0 MCP tools, ADR-0002.
- `_INSCOPE_BATCH_CYPHER` / `_FILESCOPE_BATCH_CYPHER` — the polymorphic
  templates are byte-for-byte the v1.1 versions. P1b changed only what
  indexes exist underneath them.

## Follow-ups

- Port the `CREATE CONSTRAINT` statements to FalkorDB's `GRAPH.CONSTRAINT`
  syntax so uniqueness is actually enforced (currently relies on MERGE keys).
- `parse_file` is now the largest cold-index phase (56.7 % on cgc-fork).
  Tree-sitter parsing is CPU-bound; parallelism is the obvious lever for a
  future version.

## Verdict

**Ship.** v1.2 P1b lands the real `flush_inscope` lever — five lookup
indexes that `create_schema()` had silently been failing to create since
v1.0:

- cgc-fork: `flush_inscope` −96 %, `function_calls` −93 %, total wall −50 %
  vs P1a and −82 % vs v1.0. Every parseable-source edge count identical.
- di-copilot: byte-parity proving in flight; doc updated on completion.
- 31 tests pass; `ruff` clean; schema, query core, MCP surface untouched.

c.ii (the eng review's pick) was implemented, measured, falsified, and
discarded — recorded here so it is not re-attempted. The proving gate did
its job: it caught a dead lever before it shipped.
