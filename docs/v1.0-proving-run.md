# CGA v1.0 — Proving Run

- **Date:** 2026-05-19
- **Task:** T10 — the v1.0 gate (per execution-plan decision D8)

## What this is

v1.0's pass/fail gate: index a real repository end-to-end and confirm the tools
answer correctly. Per D8, the numeric bars are *set from this first run* and
recorded here as the v1.0 baseline.

## Environment

- FalkorDB: stock `falkordb/falkordb` container (colima), `127.0.0.1:6379`.
- CGA on `main`, waves T2–T9 merged. Indexed via `cga.engine.GraphBuilder`,
  queried via `cga.query.QueryCore`. Python 3.12, Intel macOS.

## Proving repo — `cgc-fork` (the CodeGraphContext fork)

CGA indexing the codebase it was forked from — direct dogfooding. A real, mixed
Python repository (`src/`, `tests/`, `scripts/`, `organizer/`).

Precision: every spot-check resolved to the exact definition site (file +

### `index_status` & staleness — PASS (covered by the suite)

- `index_status` (`empty` → `indexing` → `indexed`) is exercised over the real
  MCP stdio protocol by `tests/test_mcp_e2e.py`.
- The mtime-floor staleness mechanism (edit a file → the next query reflects it,
  window = 1 query) is verified by
  `tests/test_mcp.py::test_staleness_refreshes_changed_file`.

### Observations

- The engine logs a `warning` per unresolved call (method calls on
  dynamically-typed receivers — `.get()`, `.run()`, builtins). Expected;
  `CGA_SKIP_EXTERNAL_RESOLUTION=true` quiets it. The 4795 `CALLS` edges are the
  *resolved* calls.
- `find_callers` counts call *sites*; `find_references` counts distinct
  referencing *entities* (it dedups multiple calls from one function) — hence
  `find_callers("debug_log")=84` vs `find_references=74`. Each is correct for
  its question.

## Scale test — `di-copilot`

Per D8, di-copilot is a scale test only: the time is recorded, it is **not** a
gate.

di-copilot is a large mixed Python + JS/TS codebase — **5017 indexable files**,
roughly 14× the cgc-fork proving repo.

| metric | value |
|---|---|
| files indexed | 5017 |
| wall time | 5396 s (~90 min) |
| throughput | 0.9 files/s |
| `Function` nodes | 19 121 |
| `Class` nodes | 3 134 |
| `CALLS` edges | 91 495 |
| `IMPORTS` edges | 17 215 |

**Correctness holds at scale.** On the 5017-file graph the tools answered
correctly: `find_symbol("main")` → `ambiguous`, 105 candidates (a `main` in
every script — the ambiguity contract firing correctly); `find_callers("main")`
→ 104 callers (the `if __name__ == "__main__"` File-level call sites);
`find_references("main")` → 104.

**Scale finding — the cold index is superlinear.** Per-file throughput fell
from 5.6 files/s (cgc-fork) to 0.9 files/s (di-copilot) — ~6× slower per file.
The bottleneck is the `CALLS`-resolution second pass: `_create_function_calls`
issues several FalkorDB round-trips per call, and di-copilot has 91 495 calls.
This is consistent with the plan's anticipated "cold index of a large repo runs
minutes" — the reason cold indexing is async. It is a **v1.1 optimization
target** (batch the call-resolution queries); it does not affect correctness or
the v1.0 verdict.

## Verdict

**v1.0 PROVEN.** CGA cold-indexes a real ~360-file Python repository in ~65 s,
and the three query tools plus `index_status` answer correctly with exact
precision against hand-checked ground truth. The architecture — transplanted
engine → FalkorDB → rewritten query core → MCP server — works end-to-end.

The di-copilot scale test confirms correctness holds on a 5000-file graph and
surfaces the one clear v1.1 priority: optimising the superlinear cold index.
