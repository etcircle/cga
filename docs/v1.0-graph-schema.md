# CGA v1.0 — Frozen Graph Schema

**Status:** FROZEN — drafted at T2 (2026-05-19) from the fork's engine; verified
against the *transplanted* `graph_builder` at the end of T3.

**Authority:** this document is the contract between the **engine** (writes the
graph) and the **query core** (reads it). The T4 query core is written against
*this document*, not against a fresh reading of `graph_builder.py`. If the
transplanted engine and this doc disagree, the T3 parser-verification test
fails and one of them is corrected — they do not silently drift.

## How this schema was derived

Extracted from the CodeGraphContext fork's `tools/graph_builder.py`
(`create_schema` + `add_file_to_graph` + `_create_function_calls` +
`_create_inheritance_links`) and the Python parser `tools/languages/python.py`.
The fork writes a Neo4j-flavoured graph; CGA's transplanted FalkorDB Lite layer
runs the same Cypher (see the *Constraints* section for the DDL translation).

## Scope

v1.0 indexes **Python and JavaScript/TypeScript only**. The fork's engine emits
labels for ~15 languages; labels for other languages are listed under
*Deferred* — the engine still creates them if fed such a file, but v1.0's query
core does not target them and the v1.0 proving repo is Python-only.

## Graph identity

One FalkorDB graph per `repo_id` (git remote URL, else hashed repo-root path).
Every label and relationship below lives inside a single repo's graph.

## Node labels — deferred (non-v1.0 languages)

`Trait`, `Macro`, `Struct`, `Enum`, `Union`, `Record`, `Annotation`,
`Property` — each keyed `(name, path, line_number)` UNIQUE. The engine creates
them for Rust/C/C++/C#/Java/etc.; v1.0's query core ignores them. Not part of
the v1.0 contract — may change in v1.1.

## Relationships

All directed: `(A)-[:REL]->(B)`.

### `CONTAINS` — structural nesting *(no properties)*
- `(Repository)-[:CONTAINS]->(Directory | File)`
- `(Directory)-[:CONTAINS]->(Directory | File)`
- `(File)-[:CONTAINS]->(Function | Class | Variable | Interface | …deferred labels)`
- `(Class)-[:CONTAINS]->(Function)` — methods
- `(Function)-[:CONTAINS]->(Function)` — nested functions

### `HAS_PARAMETER` *(no properties)*
- `(Function)-[:HAS_PARAMETER]->(Parameter)`

### `IMPORTS`
- `(File)-[:IMPORTS]->(Module)`
- properties: `line_number` (int, opt), `alias` (str, opt),
  `imported_name` (str, opt — JS/TS only)

### `CALLS` — caller invokes callee
- `(Function | Class | File)-[:CALLS]->(Function | Class)`
- A call to a `Class` resolves to that class's `__init__` / `constructor`
  `Function` when one exists; otherwise the edge points at the `Class` itself.
- properties: `line_number` (int — call site), `args` (list[str]),
  `full_call_name` (str)
- **Direction is caller → callee.** `find_callers(X)` walks
  `MATCH (caller)-[:CALLS]->(x {name: X})`.

### `INHERITS` *(no properties)*
- `(Class)-[:INHERITS]->(Class)` — child → parent

### `IMPLEMENTS` *(no properties)*
- `(Class)-[:IMPLEMENTS]->(Interface)` — TypeScript / C#

### `INCLUDES` — deferred *(Ruby mixins)*
- `(Class)-[:INCLUDES]->(Module)`. Not in the v1.0 contract.

## Constraints & indexes — and the FalkorDB caveat

The fork issues Neo4j DDL: UNIQUE constraints on the identity tuples above,
`.lang` indexes (`function_lang`, `class_lang`, `annotation_lang`), and a
`CREATE FULLTEXT INDEX code_search_index`.

The transplanted `database.py` (`_translate_schema_query`) rewrites this DDL
for FalkorDB:
- single-property UNIQUE constraints → translated;
- **composite-key constraints downgrade to plain indexes** (FalkorDB has no
  composite uniqueness — identity tuples are enforced by the engine's `MERGE`,
  not by the DB);
- **`CREATE FULLTEXT INDEX` silently becomes `RETURN 1` — a no-op.**

**Consequence for the T4 query core:** there is no fulltext index on FalkorDB.
`find_symbol` must match on the `name` property directly (exact, or a
`CONTAINS` substring match in Cypher) — **not** via `db.idx.fulltext`. The
v1.1 `code_search` tool will need its own mechanism.

## Notes for the T4 query core

1. **Symbol identity is `(name, path, line_number)`.** `name` is not unique;
   a bare-name lookup may hit several nodes → return candidates with
   `status:ambiguous` per the runtime error/empty contract.
2. **`CALLS` is the spine of `find_callers` and `find_references`.** A caller
   may be a `File`, `Function`, or `Class`; return enough of it (label, name,
   path, line_number) for an agent to act.
3. **Constructor calls:** a `CALLS` edge into a `Function` named
   `__init__`/`constructor` is a class instantiation — `find_references` on a
   `Class` should also walk `CALLS` into its constructor.
4. **`Module` is global.** Modules collide across repos by name; harmless in
   v1.0 (one graph per repo) but never assume `Module` is path-scoped.
5. **`lang` filtering:** v1.0 success is measured within one language; the
   query core must be able to filter by `lang`
   (`python`, `javascript`, `typescript`).
6. The render seam emits FalkorDB Cypher directly (decision D2). Everything
   above is plain Cypher — none of it needs the v1.1 dialect IR.

## Freeze / verification

This schema is **frozen for v1.0** as of T2. T3 transplants the engine that
produces it; the T3 parser-verification test indexes the proving repo and
diffs the resulting graph (labels, relationship types, key properties) against
this document. T4 is built strictly against this doc. Any change after T3
requires re-opening the freeze and re-checking T4.
