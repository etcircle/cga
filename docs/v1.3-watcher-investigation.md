# v1.3 ‚Äî live watcher: the ‚àí5,876 deficit was a test-harness artifact

- **Status:** RESOLVED 2026-05-23 (was: SHELVED 2026-05-22 ‚Äî the shelving was
  based on a flawed measurement).
- **TL;DR:** The watcher's warm whole-edge re-index is byte-identical to cold
  on cgc-fork. On di-copilot the only remaining gap is the known
  `.cgcignore`/`.gitignore` discovery divergence (52 cold-only files, ~15%
  CALLS deficit) ‚Äî the design doc ¬ß5 issue, ordinary work, not architectural.
  The earlier "watcher cannot reproduce the cold-index graph, even with a
  full rebuild" finding was caused by an unresolved `/tmp` path in the
  proving harness, which silently disabled cold's `.cgcignore` filtering.
  The watcher was fine; the harness was lying.

## Resolution (2026-05-23)

Controlled A/B via `_local-archive/watcher-shelved/phase0_pin_5876.py` on
cgc-fork. The single difference between runs is whether the scratch repo path
is resolved (`repo = repo.resolve()`) before being passed to
`build_graph_from_path_async`:

| run | cold files | cold CALLS | watcher files | watcher relink CALLS | diff |
|---|---:|---:|---:|---:|---:|
| Run 1 ‚Äî `/tmp/‚Ä¶` (= original spike conditions) | 294 | 12,535 | 166 | 5,461 | ‚àí7,074 (reproduces the original deficit class) |
| Run 2 ‚Äî resolved path (= production conditions) | 166 | 4,795 | 166 | 4,795 | **0 missing, 0 extra, byte-identical** |

Run 2 measurements:

- Identical file sets (0 cold-only, 0 watcher-only).
- Identical imports_maps (0 key/order/membership divergence).
- Watcher relink == cold on edge-identity multiset (caller path+name+line,
  callee path+name+line, rel line_number, full_call_name).
- Strict and non-strict watcher relink agree (4,795 == 4,795) ‚Äî no flush
  chunk failures.
- Run A (cold inputs, strict, wipe+rebuild) reproduces cold exactly ‚Üí engine
  is sound and the wipe-and-rebuild relink primitive is correct.

## Mechanism

`build_graph_from_path_async` (`graph_builder.py:1543-1556`) filters files
against `.cgcignore` via `f.relative_to(path.resolve())`. When the path
passed in is not already resolved (e.g. macOS `/tmp/...`, which symlinks to
`/private/tmp/...`), `relative_to` raises `ValueError` and the
`except ValueError: filtered_files.append(f)` branch silently **keeps** the
file ‚Äî disabling `.cgcignore` filtering entirely for the cold index.

The original spike (`_local-archive/watcher-shelved/cga_proving_watcher.py`)
cold-indexed `/tmp/.../repo`, so its "cold baseline" indexed
`docs/site/...min.js` + `website/...` ‚Äî files `.cgcignore` explicitly
excludes ‚Äî pulling in ~7,700 spurious CALLS, predominantly from a single
minified JS bundle. The watcher's own discovery (`_get_supported_files` ‚Üí
`_should_ignore`) used `/tmp` paths consistently and didn't hit the bug, so
it correctly honored `.cgcignore` + `.gitignore`. The harness compared the
watcher's **correct** graph against cold's **bug-inflated** graph and
reported the gap as a "watcher deficit."

Production never hits this: `CGAMCPServer.__init__` resolves `repo_root`
before indexing.

## What this proves

- The shelving rationale is void. The warm whole-edge re-index ‚Äî the design
  doc's Phase 1 floor ‚Äî works.
- The design doc's ¬ß2 ("why the watcher produces half the CALLS") explains a
  symptom that doesn't exist. All five divergence points are moot for this
  symptom.
- H2 (silent flush failures from `is_cold_index=False`) is not in play here.

## di-copilot confirmation (2026-05-23)

Re-running the same diagnostic against di-copilot (2,973-parsed-file cold
index, 91,631 CALLS ‚Äî about an order of magnitude larger than cgc-fork):

| run | files | CALLS | vs cold |
|---|---:|---:|---:|
| Cold (oracle) | 2,973 | 91,631 | ‚Äî |
| Run A ‚Äî cold inputs, strict | ‚Äî | 91,631 | 0 (engine sound) |
| Run B1 ‚Äî watcher inputs, strict | 2,921 | 77,535 | ‚àí14,096 |
| Run B2 ‚Äî watcher inputs, non-strict | 2,921 | 77,535 | ‚àí14,096 |

Engine sound and H2 ruled out on di-copilot too. The remaining ‚àí14,096
(~15% of cold) traces cleanly to discovery divergence: **52 cold-only files,
zero watcher-only**, carrying 3,679 raw `function_calls`. 89.7% of the
sampled missing edges have a caller file the watcher never indexed. The
cold-only files concentrate in directories di-copilot's `.gitignore`
excludes but its `.cgcignore` doesn't ‚Äî `memory/2026-*` probe scripts,
`frontend/tmp/`, `extensions/office/tmp-*`, `docs/archive/`,
`docs/artifacts/`, `backend/docs/test-runs/`. On cgc-fork those
`.gitignore`-only dirs don't exist, so divergence was zero there; on
di-copilot they exist with ~52 source files.

This is exactly the design doc's ¬ß5 issue (`discover_indexable_files`
shared) ‚Äî a discovery inconsistency, not an architectural problem.

> Harness caveat: the multiset diff query is capped by FalkorDB's default
> `RESULTSET_SIZE=10000`, so the per-file missing-edge attribution above is
> on a 10k-row sample. The total counts (cold 91,631 / B2 77,535) come from
> server-side `count(r)` and are exact. The H1-dominant attribution holds
> at 89.7% on the sample, and the M1 file-set divergence (52 cold-only) is
> exact.

## Genuine bugs that remain (none of them existential)

1. **Cold-path `.cgcignore`-disabled-on-unresolved-path** (the bug that
   caused this entire investigation). Fix: `path = path.resolve()` at the
   top of `build_graph_from_path_async`, or drop the
   `except ValueError: filtered_files.append(f)` keep-branch and let it fail
   loudly. Production dodges this via MCP resolving, but it's a latent
   footgun and it corrupted every proving run.
2. **Discovery is not unified.** Cold reads `.cgcignore` only; the watcher
   reads `.cgcignore` + `.gitignore`. On cgc-fork the `.gitignore`-only
   entries don't exist so the sets match; on di-copilot they do exist and
   produce the ‚àí14,096 deficit measured above. Design doc ¬ß5
   (`discover_indexable_files` shared) fixes it. **Upgraded to priority:
   this is the only remaining real watcher-correctness gap.**
3. **Node-replacement on delete/rename leaks `Parameter` and `Module`
   nodes.** `delete_file_from_graph` (`graph_builder.py:1198`) only
   `DETACH DELETE`s things `File-[:CONTAINS]`s, but `Parameter` is reached
   via `Function-[:HAS_PARAMETER]` and `Module` via `File-[:IMPORTS]`. Real
   bug, independent of the ‚àí5,876 narrative. Design doc ¬ß6.
4. **Incremental affected-set relink** (design doc Phase 2) is still
   genuinely unproven ‚Äî the diagnostic only exercised the full warm relink
   primitive, not the affected-set heuristic.

## What's recommended next

- Fix #1 (the cold-path resolve) ‚Äî one line.
- Build the shared `IncrementalIndexCoordinator` from the design doc, but
  start with the warm whole-edge re-index (which is essentially already
  proven). Unify discovery as part of it.
- Route the MCP `_refresh_stale_files` floor through the coordinator
  (currently it only repairs a changed file's outgoing edges ‚Äî design doc
  ¬ß7 ‚Äî which is a real freshness bug, separate from anything above).
- Phase 2 (affected-set relink) and Phase 3 (node-replacement hardening)
  are ordinary work, not architectural risk.

## Process lesson

The original investigation reasoned from measurements without verifying the
measurement methodology. The cold baseline was inflated by a harness bug
nobody questioned ‚Äî and from there, every reasoned hypothesis about flush
mechanics, imports-map order, and affected-set completeness was an attempt
to explain a phantom. Always verify the test fixture is faithful to
production before attributing surprising deltas to the system under test;
when a controlled A/B can be constructed (here: one line, `repo.resolve()`),
run it before deeper investigation.

The diagnostic that pinned this is at
`_local-archive/watcher-shelved/phase0_pin_5876.py`.

## Original investigation (2026-05-22) ‚Äî preserved as historical record

> **Note:** every measurement in this section was taken against a
> `/tmp/...` repo copy, so the cold baseline was bug-inflated (see
> *Mechanism* above). The numbers (‚àí850 / ‚àí513 / ‚àí5,876) reflect that
> artifact, not production behavior. Kept here as the record of how the
> investigation unfolded.

### Goal

Wire up `src/cga/engine/watcher.py` ‚Äî a fully-built but orphaned
`watchdog`-based file watcher (`RepositoryEventHandler`, `CodeWatcher`) ‚Äî so
the graph updates incrementally instead of needing a full cold re-index.
CONTEXT.md / ROADMAP.md designated this as post-v1.0 work.

### What was found

A reusable proving harness (`cga_proving_watcher.py`, parked in
`_local-archive/watcher-shelved/`) cold-indexed a repo copy, drove the
watcher's incremental path through a deterministic mutation set, and
compared the result against a fresh cold index of the same final state.

Proven on cgc-fork (against the bug-inflated cold baseline):

| watcher relink mode | CALLS edges (cold index "G" = 12,526) | delta |
|---|---:|---:|
| original orphaned watcher | ‚Äî | ‚àí850 |
| affected-set redesign | 12,013 | ‚àí513 |
| full wipe-and-rebuild relink | 6,650 | ‚àí5,876 |

### Phase 0 ‚Äî the gate (original framing)

Two defects diagnosed in the orphaned watcher:

1. **`_create_all_function_calls` is non-idempotent.** Re-running it on an
   already-indexed graph appeared to add spurious CALLS edges. (Real cause
   under suspicion; not load-bearing now.)
2. **`_incremental_relink`'s affected-set is unsound.** It keyed on file
   imports; the heuristic missed callers whose CALLS edges pointed into a
   changed file (real, but a normal heuristic bug, not architectural).

### The redesign (parked)

`_incremental_relink` was rewritten with an affected set = changed files ‚à™
every file whose `function_calls[].name` or class-base names a symbol
defined (pre- or post-edit) in a changed file. Result against the inflated
baseline: ‚àí850 ‚Üí +8 ‚Üí ‚àí513 after attempting to remove a `set(...)` ordering
quirk. The remaining ‚àí513 was attributed to "the watcher path can't
reproduce the cold-index pipeline's CALLS graph."

### The (wrong) architectural finding

A full wipe-and-rebuild relink driven through the watcher's path produced
6,650 vs 12,526 ‚Äî "barely half" ‚Äî concentrated in minified/vendored JS.
This was read as "the watcher's invocation of `_create_all_function_calls`
does not reproduce the cold-index pipeline's CALLS graph ‚Äî not
incrementally, and not even with a full rebuild."

The real explanation, found by code-reading the cold path: cold and watcher
use different ignore mechanisms (cold: `.cgcignore` only; watcher:
`.cgcignore` + `.gitignore`), and on a `/tmp` repo copy cold's `.cgcignore`
filtering silently no-ops via the `relative_to`/`ValueError` keep-branch
described above. The watcher was correctly excluding the minified JS the
inflated cold was incorrectly including.

### Decision (2026-05-22, since reversed)

Shelved. The proving harness's `/tmp` path + cold's `relative_to` keep-branch
made the deficit look architectural. With the harness fixed (one line:
`repo = repo.resolve()`), the watcher relink is byte-identical to cold.
