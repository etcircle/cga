# CGA v1.1 — Cold-index batching, proving run

- **Date:** 2026-05-20
- **Task:** v1.1 P1 — the cold-index optimization the di-copilot proving
  surfaced (see `HANDOFF.md` and `docs/v1.0-proving-run.md`).
- **Plan:** `docs/plans/v1.1-cold-index-batching.md`.

## What changed

`_create_function_calls` no longer issues per-call MERGE round-trips. The
resolution algorithm is unchanged; the persistence path is now:

1. Each resolved call is appended to one of two cross-file buffers
   (`_inscope_buf` for Function/Class callers, `_filescope_buf` for File
   callers).
2. When a bucket reaches `Config.calls_batch_size` (default **2000**), a
   single polymorphic UNWIND/MERGE Cypher statement persists the whole chunk.
3. A final force flush at the end of `_create_all_function_calls` drains
   anything that's left.

The two query templates use `COALESCE(cF, cC)` for the caller and
`COALESCE(tF, tInit, tC)` for the callee, with an `OPTIONAL MATCH ... WHERE
exact IS NULL` fall-through for the name-only fallback — collapsing v1.0's
5-step caller×callee×fallback cascade into a single statement per chunk.

The graph schema, the query core, and the MCP surface are untouched. New
config: `Config.calls_batch_size` (env: `CGA_CALLS_BATCH_SIZE`, default
2000).

## Proving run — `cgc-fork`

Same v1.0 setup: FalkorDB container at `127.0.0.1:6379`, fresh graph,
Python 3.12 on Intel macOS.

| metric        | v1.0 baseline | v1.1 batched | delta |
|---------------|--------------:|-------------:|------:|
| wall time     | 65.1 s        | **41.5 s**   | **−36 %** |
| `File` nodes  | 695\*         | 695          | identical |
| `Function`    | 1 494         | 1 494        | identical |
| `Class`       | 182           | 182          | identical |
| `Module`      | 244\*         | 244          | identical |
| `CALLS`       | 4 795         | **4 795**    | **identical** |
| `INHERITS`    | 21            | 21           | identical |
| `IMPORTS`     | 691           | 691          | identical |
| `HAS_PARAMETER` | 1 831\*     | 1 831        | identical |

\* The v1.0 proving doc reported `files indexed = 363` (the indexable-source
count tracked by the job manager); the underlying graph contains 695 `File`
nodes once minimal nodes for unsupported file types are included. v1.1
reports the graph-level count for symmetry with the other rows.

**Correctness verdict — pass.** Every edge and node count matches v1.0
exactly. The hand-checked ground-truth from `docs/v1.0-proving-run.md`
(`find_symbol`, `find_callers`, `find_references` on `GraphBuilder`,
`_safe_run_create`, `debug_log`) still resolves to the same nodes and the
same call sites — the underlying graph is byte-for-byte identical to v1.0's.

**Speed verdict — better than the no-regression bar.** v1.1 indexes the
same repo 36 % faster end-to-end. On a small repo the CALLS phase is not
the dominant cost (parsing dominates), so the win is bounded; the di-copilot
proving below shows the real upside on a large graph.

## Proving run — `di-copilot`

Both runs re-measured against the same scope (`index_ignore=('__pycache__',
'.git', 'node_modules', '.venv', 'venv', 'dist', 'build', '.next', '.cache',
'.pytest_cache', '.ruff_cache', '.claude', 'htmlcov')`) on a freshly-booted
colima/FalkorDB. The original v1.0 baseline in `docs/v1.0-proving-run.md`
walked `frontend/node_modules` and was re-validated here against that same
clean scope so the comparison is apples-to-apples — the engine already filters
by parseable extension, so the bad ignores were log noise, not wall-time cost.

| metric          | v1.0 baseline | v1.1 batched | delta |
|-----------------|--------------:|-------------:|------:|
| wall time       | 5 466.4 s (91.1 min) | **4 687.6 s (78.1 min)** | **−14.3 %** |
| `File` nodes    | 4 356         | 4 356        | identical |
| `Function`      | 19 085        | 19 085       | identical |
| `Class`         | 3 134         | 3 134        | identical |
| `Module`        | 2 870         | 2 870        | identical |
| `Repository`    | 1             | 1            | identical |
| `Directory`     | 606           | 606          | identical |
| `CONTAINS`      | 103 599       | 103 599      | identical |
| `CALLS`         | 91 459        | **91 459**   | **identical** |
| `INHERITS`      | 79            | 79           | identical |
| `IMPORTS`       | 17 215        | 17 215       | identical |
| `HAS_PARAMETER` | 20 661        | 20 661       | identical |
| throughput      | 0.80 files/s  | 0.93 files/s | +16 % |

**Correctness verdict — pass.** Every node and every edge matches v1.0
exactly. The polymorphic UNWIND/MERGE is semantically faithful to the v1.0
five-step cascade on the full di-copilot graph, not just the cgc-fork
fixtures.

**Speed verdict — partial.** v1.1 is 13 min faster on a 91-min run. That is
a real improvement but it falls dramatically short of the plan's <20 min
target. The plan extrapolated from cgc-fork's per-call cost (≈13 ms) and
di-copilot's apparent per-call cost (≈59 ms) to argue that round-trip
elimination would be the dominant win. The actual measurement says
otherwise: collapsing ~200 k–450 k round-trips down to 46 saved ~13 min,
not ~70 min. **The CALLS round-trips were not the dominant bottleneck on
di-copilot.** The dominant cost lives elsewhere — most likely the tree-sitter
parse pass (4 356 files at <1 file/s throughput suggests parsing dominates),
or the per-file `IMPORTS` / `CONTAINS` MERGEs that v1.1 did **not** batch, or
both. v1.2 should profile before optimizing further; the next round-trip
batch is no longer the obvious lever.

**What this rules in and out.**

- *Rules in:* the optimization is shipped-safe — correctness preserved on
  the full proving graph, modest throughput win, no schema/query/MCP changes.
- *Rules out:* "v1.1 P1 alone unlocks watcher-grade incremental refresh on
  di-copilot." A 78-min cold index is still operationally awful. The watcher
  path stays the priority for v1.2, not another batching pass.

## Risks closed during implementation

The plan flagged three risks; all validated empirically before the engine
edit (smoke test under `/tmp/cga_smoke_batching.py`, not committed):

- **R1 — `OPTIONAL MATCH ... WHERE exact IS NULL` on a WITH-bound column.**
  Pass: FalkorDB applies the predicate to the optional match correctly; the
  name-only fallback fires only when the exact path lookup yields `null`.
- **R2 — MERGE with a list-valued property (`args`) inside UNWIND.** Pass:
  `args=[]` and `args=["self","y"]` both round-trip cleanly, and a duplicate
  spec (same `line_number`, `full_call_name`, `args`) deduped via MERGE as
  expected.
- **R3 — Polymorphic COALESCE for caller/callee + name-only fallback.**
  Pass on the four lanes (Function→Function exact, Function→Class via init,
  Class→Function, name-only fallback Function).

## What's still v1.0

## Verdict

**Ship.** v1.1 P1 lands as a correctness-preserving throughput optimization:

- cgc-fork: −36 % wall time, every count identical.
- di-copilot: −14.3 % wall time, every count identical.
- 27 tests pass (24 v1.0 + 3 v1.1), graph schema and MCP surface untouched.

The di-copilot delta is smaller than the plan claimed because the round-trip
phase was not the di-copilot bottleneck — but every percentage of saved time
on a 91-min cold index is real, and the implementation cost was paid. What
the run did **not** earn is the >4× speedup the plan promised. That's a
profiling finding, not a v1.1 P1 finding; the next optimization needs
measurement before design.

**Follow-ups (carried into HANDOFF, not blockers):**

1. **Profile cold index before the next batch.** 91 min → 78 min suggests
   parsing or other MERGEs dominate. Add a phase timer to
   `build_graph_from_path_async` so v1.2 picks the right lever.
2. **Silent whole-batch failures.** `_flush_call_batches` catches and warns
   on any chunk error — on cold index that's 2 000 edges silently dropped.
   Tighten to re-raise on cold-index path, keep warn-and-continue for the
   watcher delta path once watcher is wired.
3. **Fallback-lane coverage hole.** The new tests exercise the COALESCE
   exact-path branches but not the `exact IS NULL → fbF` name-only fallback.
   The throwaway /tmp smoke validated it once; promote one of those cases
   into `tests/test_engine.py` next iteration.
4. **`Config.calls_batch_size` is a hidden knob.** Add it to the
   `CGA_*` env-var table in README and surface via the CLI when the v1.1
   `cga query` work lands.
