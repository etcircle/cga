# CGA v1.2 P1a — add-file batching, proving run

- **Date:** 2026-05-21
- **Task:** v1.2 P1a — apply the v1.1 buffered-flush pattern to the
  per-file MERGEs `add_file_to_graph` was issuing one round-trip at a time
  (Files, Directories, per-label CONTAINS, parameters, modules,
  nested-functions, JS/TS/Python imports, class-contains).
- **Driver:** the di-copilot P0.5 row in `docs/v1.2-profile.md` pinned
  `add_file_to_graph` at 23.9 % of total wall on di-copilot and 24.4 % on
  cgc-fork — the only phase with a stable per-file ratio across scales,
  highest-confidence batch target on the board.

## What changed

The cold-index code path in `build_graph_from_path_async` no longer calls
`add_file_to_graph` per-file. Instead it routes through two new methods on
`GraphBuilder`:

1. `_buffer_file_for_batch(file_data)` — pure Python, no DB round-trips.
   Translates one parsed file into rows for every UNWIND payload the
   batched flush will emit. Mirrors `add_file_to_graph`'s node/edge set
   row-for-row so the graph is byte-identical to the per-file path.
2. `_flush_file_batches(session)` — drains the buffers via UNWIND/MERGE
   queries in dependency order: Files first, then Directories
   depth-by-depth, then File-CONTAINS edges, then per-label
   (Function/Class/Variable/Interface) MERGE + CONTAINS, then Parameters,
   Modules, nested-functions, JS/Python imports, and class-CONTAINS.

Buffers live on instance state and are cleared at cold-index entry. The
single-file callers (`update_file_in_graph` for the watcher delta path
and MCP `_index_one_file`) keep using the legacy `add_file_to_graph`
method unchanged — two paths, one for batching N files, one for the
watcher's N=1.

Each per-file MERGE that was a separate round-trip in v1.0 / v1.1 becomes
one row in a cross-file UNWIND payload, drained every
`Config.add_file_batch_size` (default 500) files plus a final force
flush before the inheritance / function-calls passes start.

The graph schema, the query core, and the MCP surface are untouched. New
config: `Config.add_file_batch_size` (env: `CGA_ADD_FILE_BATCH_SIZE`,
default 500).

`function_calls` is not touched by P1a, yet it dropped 22.6 % per file
between the two runs — almost exactly the 23.6 % that the P1a-targeted
`add_file_to_graph` dropped. Every FalkorDB-write phase moved together;
`parse_file` (CPU-bound, no DB) stayed flat at +1 %. That signature is
**FalkorDB run-to-run variance**, not a P1a effect: the di-copilot
P0.5 and P1a runs were measured at different times against the shared
`cga-falkordb` container, and the write path was ~22-23 % cheaper during
the P1a run regardless of the engine code.

So the di-copilot wall-time delta (4 134 s → 3 249 s, −21 %) **cannot be
attributed to P1a** — it is confounded with the DB-speed shift. The
di-copilot run's job here was the byte-parity gate, and it passed it.
The clean, isolated measurement of the P1a effect is cgc-fork, where
`function_calls` stayed flat (10.33 s → 10.41 s, +0.8 %) while
`add_file_to_graph` collapsed 69 %. Same-run isolation is the proof; a
cross-run di-copilot comparison is not.

**What di-copilot does establish:** correctness at scale (byte parity),
and that `add_file_to_graph` in absolute terms still dropped (987 s →
755 s) — directionally consistent with cgc-fork even if the magnitude is
not cleanly separable from noise. **What it does not establish:** a
specific di-copilot speedup number. We do not claim one. The v1.1 doc
extrapolated a di-copilot win from a cgc-fork ratio and was wrong by 4×;
this doc states the cgc-fork win (clean, −69 % on the phase) and the
di-copilot result (parity ✓, speedup not isolable) separately.

## Risks closed during implementation

- **R1 — Directory MERGEs must be depth-ordered.** Cypher `MERGE` rows
  inside one `UNWIND` are independent — a depth-N MATCH cannot see a
  depth-(N-1) Directory created in the same UNWIND batch. Mitigated by
  grouping the buffered dir entries by depth and flushing depth-0
  (parent=Repository), then depth-1 (parent=Directory), etc., each in
  its own UNWIND. Existing fixture tests + the new chunked-add-file test
  validate.
- **R2 — Cross-file dedup of Directory rows.** Many files share
  ancestor directories; `add_file_to_graph` used to handle the
  duplication implicitly per-file. The buffered path dedupes via
  `_file_batch_dir_seen: set[(parent_path, current_path)]` so a single
  Directory MERGE row is emitted across all files that share it.
- **R3 — Repository node assumption.** The per-file path defensively
  re-`MATCH`'d the Repository node and fell back to `file_data['repo_path']`
  if absent. The batched path trusts that `add_repository_to_graph` at
  the start of `build_graph_from_path_async` created the Repository, and
  computes `relative_path` from `file_data['repo_path']` directly. One
  fewer round-trip per file; behavior is unchanged because the
  Repository is always created.
- **R4 — File-contains parent label split.** A File's immediate parent
  is either a Repository (file at repo root) or a Directory (file under
  one or more subdirs). The v1.0 path matched the parent label by
  F-string interpolation per-call; the batched path splits the buffer
  into `file_contains_repo` and `file_contains_dir` and flushes each
  with its own labeled MATCH.

## Carry-overs from v1.1 follow-ups (handoff #1, #2, #4)

Three v1.1 follow-ups closed in the same PR because they live in the
same code path:

- **`_flush_call_batches` re-raises on the cold-index path.** v1.1's
  warn-and-continue silently dropped up to `calls_batch_size` (2 000)
  CALLS edges per chunk failure — exactly the regression the byte-parity
  gate is meant to catch. The new `is_cold_index=True` argument re-raises
  on the cold path so chunk failures surface immediately. The watcher
  delta path (`is_cold_index=False`) keeps warn-and-continue so a single
  bad chunk does not knock the watcher out of service.
- **`asyncio.sleep(0.01)` → `asyncio.sleep(0)` on the cold path.** The
  v1.2 P0 profile showed the 10 ms yield was ~11 % of cgc-fork wall time
  and ~1 % of di-copilot wall time — pure overhead with no observable
  cooperation benefit on a single-cold-index process. `asyncio.sleep(0)`
  still gives the event loop a chance to run sibling tasks.
- **`Config.add_file_batch_size`** is the third tunable knob alongside
  `Config.calls_batch_size` (v1.1) and `Config.profile_phases` (v1.2 P0).

## What's still v1.0

- Graph schema (`docs/v1.0-graph-schema.md`) — frozen, no changes.
- Query core (`cga.query.QueryCore`) — same Cypher, same contract.
- MCP server + the four v1.0 tools (`find_symbol`, `find_callers`,
  `find_references`, `index_status`) — untouched.
- ADR-0002 (FalkorDB runs as a server) — still authoritative.
- The polymorphic 5-`OPTIONAL MATCH` `_INSCOPE_BATCH_CYPHER` and its
  file-scope twin — P1b's lever, not P1a's. The `OPTIONAL MATCH (fbF)
  WHERE exact IS NULL` name-only fallback that the di-copilot P0.5 row
  identified as the dominant cost stays exactly as-is here, gated by the
  new `test_v1_2_name_only_fallback_lane_via_inscope_batch` regression
  test.

## Verdict

**Ship.** v1.2 P1a lands as a correctness-preserving throughput
optimization targeting the second-largest cold-index phase:

- cgc-fork (the clean measurement): −28.6 % total wall vs v1.2 P0+P0.5,
  **−64 % vs v1.0**. `add_file_to_graph` phase share 24.4 % → 10.6 %, a
  69 % reduction on the phase, with `function_calls` confirmed flat
  (10.33 s → 10.41 s) — same-run isolation of the P1a effect.
- di-copilot: every node and edge count byte-identical to v1.0 / v1.1
  (PASS gate #4). Wall dropped 4 134 s → 3 249 s but the delta is
  confounded by FalkorDB run-to-run variance — `function_calls`, which
  P1a does not touch, dropped a near-identical 22.6 %. No di-copilot
  speedup number is claimed; the di-copilot run's deliverable is the
  parity gate, and it passed.
- 30 tests pass (27 v1.1 baseline + 3 v1.2 P1a). `ruff` clean.
- Graph schema, query core, MCP surface untouched.

The remaining wall budget on cgc-fork is dominated by `function_calls`
at 44.8 % (`flush_inscope` 89.6 % of that). P1b — the c.ii
fallback-bucket split — is the next lever.
