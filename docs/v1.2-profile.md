# CGA v1.2 P0 — Cold-index phase profile

- **Date:** 2026-05-20
- **Task:** v1.2 P0 — retire the v1.1 follow-up #1 ("profile cold index before
  the next batch") with concrete phase-time numbers.
- **Code:** `Config.profile_phases` + `_emit_phase_profile` in
  `src/cga/engine/graph_builder.py`. Enable with `CGA_PROFILE_PHASES=true`.

## What's instrumented

`build_graph_from_path_async` accumulates wall time per phase and writes a
breakdown to stderr when `Config.profile_phases` is on. Phases:

| phase | what it covers |
|---|---|
| `repository_node` | `add_repository_to_graph` MERGE |
| `discovery` | `rglob`, `index_ignore` filter, `.cgcignore` matcher |
| `pre_scan_imports` | `_pre_scan_for_imports` (builds the imports map) |
| `parse_file` | tree-sitter parse + Python AST walk per file |
| `add_file_to_graph` | per-file MERGEs: Files, Functions, Classes, Modules, IMPORTS, CONTAINS, parameters |
| `minimal_file_node` | placeholder File nodes for unparseable extensions |
| `inheritance_links` | `_create_all_inheritance_links` |
| `function_calls` | v1.1-batched CALLS resolution + UNWIND/MERGE flush |
| `unaccounted` | `total - sum(phases)` — sleeps, I/O waits, GC, etc. |

Off by default (no per-file overhead). When on, the timing reads add an
estimated <0.5 % per phase boundary.

(2 973 here is the parsed-file count the profile uses as denominator; the

### What changed from cgc-fork

The phase ratios moved hard. Per-file cost in DB-heavy phases exploded
while non-DB phases barely changed:

| phase | cgc-fork ms/file | di-copilot ms/file | growth |
|---|---:|---:|---:|
| parse_file | 40.01 | 69.41 | 1.7× |
| add_file_to_graph | 47.80 | 341.26 | **7.1×** |
| **function_calls** | 62.24 | 942.07 | **15.1×** |

Tree-sitter parse grew 1.7× (mostly larger files, expected). The two
DB-heavy phases grew 7-15× per file. **The dominant cost is graph-size
scaling of MERGE planning + execution on a populated FalkorDB graph, not
round-trip count, not file count.** This is the di-copilot signal the
v1.1 doc said we needed before picking the v1.2 P1 lever.

### Pivot from the cgc-fork-only verdict

The cgc-fork breakdown above tilted toward option (b) (batch
`add_file_to_graph`). The di-copilot data **walks that decision back**.
Per-file impact of attacking the two top phases:

- **Option (b) — `add_file_to_graph` batching:** if we get a v1.1-shaped
  50 % cut, that saves 12 % of total wall (~8 min of 69). Real but
  bounded.
- **Going after `function_calls` again:** even a 30 % cut here saves
  20 % of total wall (~14 min of 69). The bigger lever — but v1.1
  *already* batched this phase, so the remaining 67.2 % isn't
  round-trip count, it's something else inside
  `_create_all_function_calls`.

### The new question: what's actually slow inside `function_calls`?

`function_calls` covers three sub-phases v1.2 P0 didn't split:

1. The per-file Python resolution loop in `_create_function_calls` —
   building `called_name`, resolving `imports_map` / `local_names` /
   `local_imports`, appending to `_inscope_buf` / `_filescope_buf`.
2. `_flush_call_batches` threshold flushes (mid-loop, batches of 2000).
3. `_flush_call_batches` force flush at end.

Inside (2) and (3), each `session.run(_INSCOPE_BATCH_CYPHER, batch=head)`
runs a polymorphic 5-OPTIONAL-MATCH query against the live graph. As
the graph grows, that query's per-row cost grows even though the chunk
size stays at 2 000. v1.1's batching cut *round-trips* but not *per-row
graph-size cost*.

## Revised decision for v1.2 P1

**Don't pick yet.** Add a sub-profile inside `function_calls` (v1.2 P0.5)
that splits:

- `function_calls.resolve` — Python work in `_create_function_calls`
- `function_calls.flush_inscope` — wall time inside the in-scope UNWIND/MERGE
- `function_calls.flush_filescope` — wall time inside the file-scope UNWIND/MERGE

This is ~20 lines of additional instrumentation and costs nothing on the
default path. Re-run on di-copilot, look at the sub-split, **then** decide
between:

- **(b′)** — batch `add_file_to_graph` (locked in as a follow-on regardless
  — second-biggest phase, known pattern, low risk).
- **(c)** — simplify the polymorphic COALESCE in `_flush_call_batches`:
  split the in-scope template into Function-caller and Class-caller
  templates (two simpler queries, two round-trips per chunk instead of
  one); drop the `exact IS NULL → fbF` name-only fallback to its own
  third flush. If FalkorDB's planner is paying the cost of optionality
  in the 5-MATCH chain, this should bite.
- **(d)** — push resolution work out of the per-file `for` loop into a
  pre-built lookup table that the flush queries consume directly,
  avoiding the per-chunk WHERE re-evaluation against the growing graph.

The right choice depends on whether resolve or flush dominates inside
`function_calls`. The sub-profile decides.

add_file_to_graph          7.94 s   24.4%    47.80 ms/file
minimal_file_node          2.08 s    6.4%    12.53 ms/file

(`ms/file` denominator is the 166 parsed-file count, not all 364 — so

## Findings

### 1. `function_calls` is still the largest phase even after v1.1 batching

10.33 s of 32.54 s — 31.7 % — is the v1.1-batched CALLS pass. The v1.1 plan
collapsed 200 k–450 k round-trips down to ~46 chunks. That win is locked in.
What's left in this bucket isn't round-trip count — it's the cost of the
polymorphic COALESCE query *per chunk*. Each batch evaluates five
`OPTIONAL MATCH` clauses and a `WHERE caller IS NOT NULL AND called IS NOT
NULL` filter against an N-node graph; the plan-time and per-row cost may
grow with N.

**Next probe:** instrument *inside* `_flush_call_batches` to split flush
wall-time from resolution wall-time. If flush dominates, the polymorphic
query shape is the lever (split the cascade into 2 simpler queries; drop
the name-only fallback to an explicit second flush). If resolution
dominates, the per-call Python loop is the lever.

### 2. `add_file_to_graph` is the obvious next batching target (option b from the v1.1 doc)

7.94 s, 24.4 %, 47.80 ms/file. This phase holds the per-file MERGEs for
`File`, `Function`, `Class`, `Module`, `IMPORTS`, and `CONTAINS` — and
unlike CALLS, none of these were batched in v1.1. Each indexed file
issues many MERGEs serially. cgc-fork has 8 042 `CONTAINS` and 691
`IMPORTS` — di-copilot has 103 599 / 17 215. Batching this phase by the
same UNWIND/MERGE pattern v1.1 used should move the needle.

**Caveat — the v1.1 anti-pattern:** the same logic ("CALLS round-trips
dominate") under-delivered on di-copilot because the per-call cost wasn't
just round-trip overhead. Before committing to batching
`add_file_to_graph`, **re-profile against di-copilot** to confirm the same
phase ratio at scale. The cgc-fork breakdown is suggestive, not
definitive; the di-copilot run may show a very different shape.

### 3. `parse_file` (tree-sitter) is third at 20.4 %

40 ms/file on cgc-fork. Hard to optimize without rewriting against
different grammars or running parsers in parallel. **Low-priority lever**
unless di-copilot's TypeScript-heavy frontend changes the mix.

### 4. `asyncio.sleep(0.01)` per file is ~11 % of cgc-fork wall time

`build_graph_from_path_async` does `await asyncio.sleep(0.01)` after
every file. 364 files × 10 ms = 3.64 s of pure overhead — almost the
entire 4.12 s `unaccounted` bucket. The sleep exists to yield the event
loop so concurrent watchers/queries don't starve; `await asyncio.sleep(0)`
or removing it entirely on the cold path would recover all of this. On
di-copilot at 4 356 files × 10 ms = 43.6 s (about 1 % of the 4 687 s wall),
so it's a cgc-fork-shaped win and a di-copilot-shaped rounding error —
worth doing on the merits, not as a v1.2 P1 lever.

## Decision for v1.2 P1 (superseded by the di-copilot data above)

The cgc-fork-only verdict was option (b) — batch `add_file_to_graph`. The
di-copilot row pivoted that to **sub-profile inside `function_calls`
first** (v1.2 P0.5), then decide between (b′), (c), or (d). See the
"Revised decision for v1.2 P1" section above.
