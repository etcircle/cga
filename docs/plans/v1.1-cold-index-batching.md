# v1.1 P1 — batch the CALLS round-trips

## What we're fixing

The di-copilot proving run surfaced one clear bottleneck: the second pass that
creates `CALLS` edges (`GraphBuilder._create_function_calls`) issues several
FalkorDB round-trips **per call site**. On the proving repos:

| repo       | files | `CALLS` edges | wall time | per-call cost\* |
|------------|-------|---------------|-----------|-----------------|
| cgc-fork   | 363   | 4 795         | 65 s      | ≈ 13 ms         |
| di-copilot | 5 017 | 91 495        | ~5 400 s  | ≈ 59 ms         |

\* total wall-time / total CALLS — overestimates the CALLS phase but the
relative growth tells the story.

Per-call cost grew ~4.5× as the graph grew ~19×. Two compounding causes:

1. **Per-call round-trips.** Each call site triggers up to **5** sequential
   MERGE queries — the fallback cascade Function→Function → Function→Class →
   Class→Function → Class→Class → name-only Function. File-level callers get
   their own 3-step cascade. Resolved calls hit on the first one; unresolved
   calls walk the full chain.
2. **MERGE on a growing graph.** Each round-trip is bigger as more candidates
   match (FalkorDB still has to evaluate the OPTIONAL MATCHes).

On di-copilot the first cause dominates: ~91 495 calls × ~2-5 round-trips =
**roughly 200 000–450 000 round-trips** for the CALLS phase alone. Each
round-trip pays Redis-RESP framing + FalkorDB query parsing + plan
construction + MERGE locking — none of which amortize.

## The fix in one sentence

Replace the per-call cascade with **one polymorphic UNWIND/MERGE query per
batch**, chunked across files; preserve resolution priority via Cypher
`COALESCE` instead of Python `if not _safe_run_create: ...` chains.

## Design

### 1. Buffer per-call specs cross-file

`_create_function_calls` stops issuing queries. Instead it appends fully-
resolved specs to two batch buffers held on the `GraphBuilder` instance:

```python
@dataclass
class _CallEdgeSpec:
    caller_name: str | None        # None ⇒ file-level caller
    caller_path: str
    called_name: str
    called_path: str               # resolved path (best guess)
    line_number: int
    args: list[str]
    full_call_name: str
```

Two buckets, because the Cypher shape differs:

- `_inscope_buf`: `caller_name is not None` — caller is a `Function` or
  `Class` (must polymorphically resolve both ends).
- `_filescope_buf`: `caller_name is None` — caller is a `File` (resolve only
  the callee polymorphically).

Resolution stays exactly where it is today (in-memory `imports_map` +
`local_names` + `local_imports` lookups). The change is purely about **when
we hit the database**.

### 2. Flush in chunks

Two flush triggers:

- **Threshold.** When either buffer reaches `Config.calls_batch_size`
  (default **2 000**), flush that bucket.
- **End-of-pass.** `_create_all_function_calls` flushes any remainder.

Chunk size is a knob, not a guess. 2 000 keeps the largest UNWIND parameter
list under ~1 MB (with 8-field dicts) and is well within FalkorDB's
single-statement comfort zone. Tunable via `CGA_CALLS_BATCH_SIZE`.

### 3. The polymorphic Cypher

**In-scope batch** (one round-trip for `chunk` items):

```cypher
UNWIND $batch AS item
OPTIONAL MATCH (cF:Function {name: item.caller_name, path: item.caller_path})
OPTIONAL MATCH (cC:Class    {name: item.caller_name, path: item.caller_path})
WITH item, COALESCE(cF, cC) AS caller
OPTIONAL MATCH (tF:Function {name: item.called_name, path: item.called_path})
OPTIONAL MATCH (tC:Class    {name: item.called_name, path: item.called_path})
OPTIONAL MATCH (tC)-[:CONTAINS]->(tInit:Function)
  WHERE tInit.name IN ['__init__', 'constructor']
WITH item, caller, COALESCE(tF, tInit, tC) AS exact
OPTIONAL MATCH (fbF:Function {name: item.called_name})
  WHERE exact IS NULL
WITH item, caller, COALESCE(exact, fbF) AS called
WHERE caller IS NOT NULL AND called IS NOT NULL
MERGE (caller)-[r:CALLS {
  line_number:    item.line_number,
  full_call_name: item.full_call_name,
  args:           item.args
}]->(called)
RETURN count(r) AS created
```

**File-scope batch**:

```cypher
UNWIND $batch AS item
OPTIONAL MATCH (caller:File {path: item.caller_path})
OPTIONAL MATCH (tF:Function {name: item.called_name, path: item.called_path})
OPTIONAL MATCH (tC:Class    {name: item.called_name, path: item.called_path})
OPTIONAL MATCH (tC)-[:CONTAINS]->(tInit:Function)
  WHERE tInit.name IN ['__init__', 'constructor']
WITH item, caller, COALESCE(tF, tInit, tC) AS exact
OPTIONAL MATCH (fbF:Function {name: item.called_name})
  WHERE exact IS NULL
WITH item, caller, COALESCE(exact, fbF) AS called
WHERE caller IS NOT NULL AND called IS NOT NULL
MERGE (caller)-[r:CALLS {
  line_number:    item.line_number,
  full_call_name: item.full_call_name,
  args:           item.args
}]->(called)
RETURN count(r) AS created
```

That's **two query templates**, executed N times where N = `ceil(total_calls
/ batch_size)`. For di-copilot, 91 495 calls / 2 000 ≈ **46 round-trips total**
instead of 200 000–450 000.

### Semantic equivalence with v1.0

v1.0's cascade is deterministic in this order — caller-label priority then
callee-label priority, then a name-only fallback:

| step | caller | callee (exact path) |
|------|--------|--------------------|
| 1    | Function | Function           |
| 2    | Function | Class (→__init__) |
| 3    | Class    | Function           |
| 4    | Class    | Class (→__init__) |
| 5    | Function or Class | Function (name only) |

The polymorphic query expresses the same priority via column-wise `COALESCE`:

- `caller = COALESCE(cF, cC)` — Function over Class.
- `exact = COALESCE(tF, tInit, tC)` — Function over Class-with-init over Class.
- `called = COALESCE(exact, fbF)` — exact match over name-only fallback.

Trace through the four interesting cases:

| state | v1.0 picks | new picks |
|-------|-----------|-----------|
| `cF` exists, `tF@path` exists | (cF, tF) at step 1 | (cF, tF) — COALESCE ⇒ first non-null |
| `cF` exists, only `tC@path` exists | (cF, tC) at step 2 | (cF, tC) — COALESCE skips tF=null |
| only `cC` exists, `tF@path` exists | (cC, tF) at step 3 | (cC, tF) — caller falls to cC, callee tF wins |
| no path match, only fbF exists | (cF or cC, fbF) at step 5 | (caller, fbF) — exact=null ⇒ fallback fires |

Same edges, same priority, one round-trip per chunk instead of one round-trip
per call.

The **mid-cascade short-circuit** in v1.0 — "stop trying once `created > 0`" —
is also preserved: each item produces at most one `MERGE` because we
`COALESCE` to a single `caller` and a single `called` before the MERGE
clause. There's no risk of double-creating edges by running step 2 after step
1 already succeeded.

### Edge-property MERGE key

The v1.0 MERGE pattern keys on three properties:

```cypher
MERGE (a)-[:CALLS {line_number: ..., args: ..., full_call_name: ...}]->(b)
```

I keep all three in the new MERGE to preserve faithful semantics (this is
what controls dedup). If FalkorDB's MERGE on a list-valued property
(`args`) misbehaves in the UNWIND form (rare but possible — list equality
semantics vary across Cypher implementations), the fallback is to MERGE on
`(line_number, full_call_name)` only and `SET r.args = item.args`. In
practice line + full_call_name already uniquely identify a call site within a
caller; `args` rarely changes the dedup outcome. **We decide which one to
keep based on the smoke test, not by guessing now.**

## What does NOT change

- The **graph schema** (`docs/v1.0-graph-schema.md`) — same nodes, same edge
  type, same properties. Schema freeze stays intact.
- The **query core** — `find_symbol` / `find_callers` / `find_references`
  read the same edges. Test contract preserved.
- The **resolution algorithm** — every `imports_map` / `local_names` /
  `local_imports` rule in the current `_create_function_calls` is kept
  verbatim. The change is purely the persistence path.
- **`skip_external_resolution`** — when set, items with no resolved path are
  still dropped before being buffered, exactly as today.
- **MCP surface** and the four v1.0 tools — untouched.

## Open risks (validated before implementing)

These are checked in **task #2 (smoke test)** before any production code
changes:

1. **`OPTIONAL MATCH ... WHERE exact IS NULL` semantics on FalkorDB.**
   In neo4j-flavor Cypher, a `WHERE` clause attached to an `OPTIONAL MATCH`
   filters the optional pattern; if the filter rejects, the match yields
   `null`. FalkorDB should match — but verify on a 2-row fixture before
   trusting it across 90 k rows.
2. **MERGE with a list-valued property in an UNWIND.** The `args` list per
   call site is typically `["x", "y"]` or `[]`. Confirm `MERGE
   (a)-[:CALLS {args: $item.args}]->(b)` dedups correctly for repeated
   item with the same list. If not, downgrade to MERGE on `(line_number,
   full_call_name)` + `SET args`.
3. **Chunk size headroom.** UNWIND a 2 000-element list in one
   `graph.query()` call — confirm latency is bounded (target: <3 s per
   chunk on a warm graph) and no truncation/timeout issues.
4. **Plan caching.** FalkorDB caches query plans by query text. The two
   batch templates use the same text every flush, so plan compilation
   amortizes across all chunks. (v1.0 had no such locality — each cascade
   query was its own text with `$caller_name`/`$called_name`/etc. inlined
   as parameters, but the cascade structure varied per branch.)

## Implementation plan

Five tasks in TaskList; see in-repo `TaskList`. Headlines:

1. **Smoke test** the polymorphic UNWIND/MERGE on a fresh FalkorDB graph with
   a hand-built 5-row batch. Confirm the three risk items above. Throw the
   test away after.
2. **Refactor `_create_function_calls`** to push specs into
   `_inscope_buf` / `_filescope_buf` instead of issuing queries. Drop the
   `_safe_run_create` cascade. Resolution code is unchanged.
3. **Add `_flush_call_batches(session, chunk_size)`** running the two
   templates above. Wire threshold flushing into the
   per-file loop in `_create_all_function_calls` (flush when bucket size ≥
   `Config.calls_batch_size`), plus a final flush.
4. **Add `Config.calls_batch_size`** (default 2 000) + `CGA_CALLS_BATCH_SIZE`
   env override. Document in CONTEXT / README.
5. **Extend tests** (`tests/test_engine.py`) — keep the existing
   `call_helper → helper` parity check; add Function→Class-with-`__init__`,
   File→Function, and the name-only fallback paths. Tiny fixture additions
   as needed; the suite must stay live-FalkorDB-gated like today.

## Proving plan

Once tests pass:

- **cgc-fork** — re-index, confirm 363 files in **≤ 65 s** (no regression).
  Hand-checked ground truth from `docs/v1.0-proving-run.md` must still
  return identical results for `find_symbol`, `find_callers`,
  `find_references`. Edge counts (`CALLS=4795`, `INHERITS=21`,
  `IMPORTS=691`) must match exactly.
- **di-copilot** — re-index, target **< 20 min** (4.5× speedup from 90 min),
  stretch **< 10 min**. Edge counts (`CALLS=91495`, `Function=19121`, etc.)
  must match v1.0 exactly — *the optimization is about throughput, not
  correctness*.

Record both in `docs/v1.1-cold-index-batching.md` as the v1.1 baseline,
mirroring the v1.0 proving doc's structure. If both pass, rotate `HANDOFF.md`
and close out v1.1 P1.

## Non-goals for this work

Deliberately out of scope (deferred to other v1.1 priorities per HANDOFF):

- CLI client (`cga query` / `cga show`).
- `get_file_structure`, `code_search`, `get_module_overview`.
- `graph_query` escape-hatch + ADR-0001 dialect IR.
- Live `cga watch`.

The CALLS batch is meant to be a **surgical, semantically transparent**
optimization. If it adds anything beyond throughput, the diff is too big.
