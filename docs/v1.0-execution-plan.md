## What v1.0 is

The smallest slice that proves the CGA architecture end-to-end: a transplanted
indexing engine writing into FalkorDB Lite, a rewritten query core, an MCP
client, and 3 read tools + `index_status`. Proven on a small Python repo first
with numeric bars, then di-copilot as a scale test.

## Decisions locked by this review

- **D1** — v1.0-first scope (v1.1 = CLI client, rest of tools, `graph_query`).
- **D2** — defer the dialect IR to v1.1; v1.0's builder emits FalkorDB Cypher
  directly behind a render-step seam.
- **D6** — backend is **FalkorDB Lite**, not Kuzu. Verified: Kuzu is not an
  installed dependency, has zero tests, and `database_kuzu.py` is a regex
  Neo4j→Kuzu translator. FalkorDB Lite is the fork's actual tested embedded
  default (and the user's original "folklore" ask).
- **D7** — **no bespoke `cga` gateway daemon.** FalkorDB Lite already runs as an
  auto-spawned worker subprocess on a unix socket (`database_falkordb.py` +
  `falkor_worker.py`) and serves concurrent clients. ADR-0002 / ADR-0003 are
  moot. (D3 — the daemon↔client protocol — is therefore superseded.)
- **D8** — prove on a small Python-only repo first with numeric pass/fail bars;
  di-copilot is a follow-on scale test; cross-language resolution is out of v1.0.
- **A4 / amended** — cold index is async (return `{status:indexing}`, agent
  polls `index_status`); incremental re-index is synchronous inline.

## Architecture (v1.0)

```
  agent (Claude Code / Hermes)
        │ MCP over stdio
        ▼
  ┌──────────────────────┐
  │  cga MCP client      │  thin adapter
  │   ├ ToolRegistry     │  name → (schema, fn), no handler tier
  │   └ query core       │  intent methods + private builder
  │        (shared library, in-process)
  └──────────┬───────────┘
             │ FalkorDB/Redis protocol over a unix socket
             ▼
  ┌──────────────────────┐
  │ FalkorDB Lite worker │  auto-spawned subprocess, multi-client
  │  (the graph store)   │  TRANSPLANTED — database_falkordb.py + falkor_worker.py
  └──────────▲───────────┘
             │ writes the graph (index / re-index)
  ┌──────────┴───────────┐
  │ engine               │  tree-sitter parsers (Python + JS/TS) + graph_builder
  │  runs in the MCP client process on cold index (background thread).
  │  v1.1: also in the standalone `cga watch` process.    TRANSPLANTED
  └──────────────────────┘
```

The query core is a shared library imported in-process by the MCP client — "one
place" is one module, no separate process needed. FalkorDB Lite's worker is the
concurrency point; multiple `cga` client processes connect to it safely.

## v1.0 tool surface (3 + status)

`find_symbol`, `find_references`, `find_callers`, `index_status`.
Deferred to v1.1: `get_file_structure`, `code_search`, `get_module_overview`,
`graph_query`.

## Runtime contracts (v1.0)

- **Cold index — async.** The MCP client's first call on an unindexed repo
  starts `graph_builder` in a background thread (it already has
  `build_graph_from_path_async` + `JobManager`), returns `{status:indexing}`;
  the agent polls `index_status` until `indexed`, then re-issues its query.
- **Incremental re-index — inline.** A query that finds changed files via the
  mtime scan re-indexes just those files synchronously before answering.
- **Staleness — mtime scan only in v1.0.** No live watcher in v1.0 (the 24/7
  `cga watch` process is v1.1). The cheap per-query `stat` scan is v1.0's
  freshness mechanism; staleness window = one query.
- **FalkorDB Lite worker** auto-spawns on first connection (existing fork code);
  multiple `cga` client processes connect concurrently over its socket.
- **repo_id** — git remote URL, else hashed repo-root path; one FalkorDB graph
  per `repo_id`.
- **Error / empty contract** — miss → `{results:[],status:empty}`; ambiguous →
  candidates + `status:ambiguous`; unparseable file → skipped, surfaced in
  `warnings`. Tools never throw to the agent.

## Transplant build order (serial — outside-voice #15)

1. **Sever the config inversion first.** `get_config_value` is imported from
   `cli/config_manager.py` by `graph_builder.py`, `database_falkordb.py`,
   `database_kuzu.py` (dropped — see below), and `utils/debug_log.py`. `debug_log`
   is a leaf utility imported almost everywhere, so this ripples — do it first,
   carefully: config becomes a value injected at construction.
2. **Stand up the FalkorDB Lite layer standalone** — `database_falkordb.py` +
   `falkor_worker.py` + closure. Verify a graph can be written and queried.
   **Freeze the graph schema here.**
3. **Transplant the engine** — tree-sitter parsers (Python + JS/TS only),
   `graph_builder.py`, `watcher.py` (carried for v1.1; not wired live in v1.0),
   `utils/` (`debug_log`, `tree_sitter_manager`), `core/jobs.py`. Verify
   `graph_builder` writes a correct graph into FalkorDB Lite.
4. **Only after the schema is frozen** (steps 2-3): build the rewritten query
   core against that schema. Transplant and rewrite are **serial, not parallel**.

## Rewrite — new code

- **Query core** (replaces `code_finder.py`, 1,039 LOC): intent methods over a
  private builder that emits FalkorDB Cypher directly, behind a render-step seam
  (the v1.1 dialect IR slots under that seam without touching call sites). 3
  intent methods for v1.0.
- **ToolRegistry**: one registration point, `name → (schema, fn)`; no handler
  tier (the fork's ~2,344-LOC pass-through layer is not recreated).
- **MCP client**: thin adapter; runs the query core in-process; owns the
  cold-index-async flow and the mtime-floor staleness check.
- **index_status**: reports indexed / fresh / indexing-in-progress, reading
  `JobManager` state plus a FalkorDB graph-populated check.

## NOT in scope (v1.0) — deferred, with rationale

- **The bespoke `cga` gateway daemon** — dissolved (D7); FalkorDB Lite's worker
  is the server.
- **CLI client** — v1.1 (thin, once the query core is proven via MCP).
- **`graph_query` escape-hatch tool, the dialect IR, a Neo4j renderer** — v1.1.
- **The standalone 24/7 `cga watch` process** — v1.1; v1.0 uses the mtime scan.
- **Cross-language reference resolution** (Python↔TS) — a known hard problem the
  fork punts on; out of v1.0, success measured within one language.
- **Branch-aware index caching; arch-health tools; the graph explorer** — phase 2.
- **Windows support** — FalkorDB Lite is Unix + Python 3.12+; revisit later.

## What already exists (reuse, do not rebuild)

- **FalkorDB Lite worker + socket server** — `database_falkordb.py`,
  `falkor_worker.py`: fully reused, the fork's tested default.
- **Indexing engine** — `graph_builder.py`, tree-sitter parsers, `watcher.py`:
  reused (watcher carried for v1.1).
- **Async job tracking** — `core/jobs.py` `JobManager`/`JobStatus`: reused;
  `index_status` builds on it — do NOT reinvent (outside-voice #6).
- **Not reused** — `code_finder.py` (rewritten as the query core), the handler
  tier (deleted), `cli/` surface, `database_kuzu.py` (the regex translator — dropped).

## Pass/fail bars (D8)

- **Proving repo:** a small Python-only repo (~100-300 files) — pick one (a
  small workspace repo, or a vendored fixture). This is the v1.0 gate.
- Cold index of the proving repo completes in under a set ceiling (set the
  number from the first run; record it).
- `find_references` / `find_callers` within-language precision ≥ a set target
  vs a hand-checked ground truth on the proving repo.
- After a file edit, the next query reflects it (staleness window = 1 query).
- All 3 tools + `index_status` return correct terse JSON.
- **di-copilot** — indexes as a scale test; record the time; NOT a v1.0 gate.

## Failure modes

| Codepath | Realistic failure | Test? | Error handling? | Visible? |
|---|---|---|---|---|
| FalkorDB worker spawn | worker fails to start | yes | surfaced error | clear |
| Cold index (large repo) | runs minutes | yes (async path) | `{status:indexing}` | clear |
| Parser on malformed file | parse throws | yes | skip + `warnings` | clear |
| mtime scan misses a change | stale result | yes | — (known v1.0 window) | silent — accepted, v1.1 watcher closes it |
| Two clients spawn a worker | socket bind clash | yes | second worker exits | clear (benign) |
| Config severing | broken import | yes (build) | fail fast | clear |
| Query-core Cypher vs FalkorDB dialect | wrong/empty result | yes (unit, fixture graph) | — | caught in tests |

Critical-gap check: the only silent failure is the v1.0 mtime-scan staleness
window — it is a *known, accepted* v1.0 limitation closed by the v1.1 watcher,
not an unflagged gap.

## Parallelization

Mostly **serial**: config severing → FalkorDB layer → engine → (freeze schema) →
query core. One parallel lane: the MCP client adapter + ToolRegistry can be
built against a stub query core while the real query core is in progress.
`Lane A: config → FalkorDB → engine → query core (sequential)` /
`Lane B: MCP client + ToolRegistry against a stub (joins Lane A at integration)`.
